## 1. Committed: SQLite lock contention fix (`33ef2c5`)

An independent review of the pushed repo found a 5th bug: `state_store._connect()` used bare `sqlite3.connect(path)` — Python's default 5s busy timeout and rollback journal mode. Alpha and Protection are separate processes writing the same file; a write-write collision outlasting 5s would raise `database is locked` and could kill an entire `intraday_monitoring` tick (exit checks included), since the call site inside `consume_order_intents` isn't wrapped in a try/except.

**Fix:** `timeout=30.0` + `PRAGMA journal_mode=WAL` in `_connect()`. Also closed a related gap the same review flagged: `_lock_in_profit` now writes the `halted` breaker-state key immediately instead of waiting for the next sync tick, closing a ≤15-minute window where Alpha could queue a new BUY intent right after a profit lock.

**Verification discipline:** first regression test attempt was flaky — passed even against unfixed code, because per-call connections close too fast to reliably collide via thread timing. Rewrote it to assert `PRAGMA busy_timeout`/`journal_mode` directly on the connection object — deterministic, confirmed red without the fix, green with it. Suite: 407 passed at commit time.

## 2. Uncommitted: Shadow-signal evaluation pipeline

**What it is:** A measurement-only pipeline that scores already-decided trading candidates with pluggable ML/statistical signals, without touching any trading decision. First provider: Kronos-small, a small (24.7M param) open-source financial time-series forecasting model.

**Why "shadow":** every part of this reads candidates *after* they're already logged (`log_candidate()` itself has zero diff) and writes only to a new `signal_values` table that nothing in the decision path reads back. A signal is shadow, promoted, or deleted — promotion (using a signal in the risk gate) is a separate, explicit, future task; this pipeline builds no gating hooks. Documented in `CLAUDE.md` under "Signal lifecycle."

**Mandatory pre-flight gate, followed as instructed:**
- Installed torch — first attempt pulled the CUDA-bundled build (~4.5GB of unused NVIDIA packages on this GPU-less Pi); caught it, uninstalled, reinstalled the correct CPU-only wheel from PyTorch's official index.
- Vendored Kronos's model code (`analyst_layer/vendor/kronos/`, MIT license) with two documented, additive changes: a relative-import fix, and new `predict_paths()`/`generate_paths()` methods — upstream's `predict()` only returns the sample-averaged path, not the individual Monte Carlo paths needed for `p_touch_win`/`path_dispersion`.
- Ran the real latency benchmark before building anything else: first result was **666s/ticker** (full spec: 512-bar lookback, 30 paths) — 7x over the 90s budget. Stopped and asked per instructions rather than proceeding. Landed on 64-bar lookback / 30 paths ≈ **58s/ticker**, with real safety margin.

**Schema:** normalized `signal_values` child table (`candidate_id, signal_name, signal_version, metric_name, value, status`), matching the existing `agent_signal_log`/`agent_signal_detail` parent/child pattern already in this codebase — one row per metric so the uplift report's per-metric IC computation is a plain SQL join, not JSON-parsing every row.

**New files:** `analyst_layer/shadow_signals.py` (provider interface + harness), `analyst_layer/kronos_provider.py`, `scripts/kronos_shadow_signal_job.py` (manually-invoked batch job — deliberately not scheduled inside the long-running Alpha daemon, since torch's memory footprint needs to be released on process exit on a RAM-constrained Pi), `scripts/signal_uplift.py` (per-signal Spearman IC + incremental IC controlling for `screen_score`, `INSUFFICIENT SAMPLE` gate at n<300, `PROMOTE-CANDIDATE`/`DELETE-CANDIDATE` verdict at |IC|≥0.03 threshold you selected).

**Two real bugs found and fixed during this work, not just at review time:**
1. Harness used `with ThreadPoolExecutor(...)` for the per-provider timeout — `__exit__` calls `shutdown(wait=True)`, which blocks until the timed-out thread actually finishes, silently turning a 1s timeout into the full hang duration. Fixed with an unmanaged executor + explicit `shutdown(wait=False)`. Confirmed red-when-reverted.
2. `pandas.Series.corr(method="spearman")` looked scipy-free but actually imports `scipy.stats` internally — scipy isn't installed here. Replaced with a manual rank-transform + `numpy.corrcoef` implementation (mathematically identical, zero new dependency).
3. `signal_uplift.py`'s original INNER JOIN silently printed "No signal_values data yet" when real data existed but no candidate had matured to a 21-session forward return yet. Fixed to a LEFT JOIN so `INSUFFICIENT SAMPLE (n=0)` — the honest, correct state — is reported instead of a misleading "no data" message.

**Verification, real not simulated:** ran the actual batch job against the production DB (seeded one verification candidate, cleaned it up after) — real Kronos output landed in the ledger (`p_touch_win=0.0, med_ret_21d=-3.77%, path_dispersion=2.99%` for AAPL), took 59.9s matching the benchmark. Ran `scripts/signal_uplift.py` for real — correctly reports `INSUFFICIENT SAMPLE` for all 3 metrics. Confirmed via `git diff --stat` zero changes to `protection_plane.py`, `guardrails.py`, `broker.py`, or `log_candidate()` itself. 420 → 426 tests as work progressed (13 new tests for this pipeline, all in the test suite now).
